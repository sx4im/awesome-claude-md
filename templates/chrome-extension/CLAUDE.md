## Architecture Rules

- **Manifest V3 only.** Service workers replace background pages. They are event-driven and can be terminated at any time. never rely on persistent in-memory state in the background script.
- **Message passing is the only cross-context communication.** Content scripts, popup, options page, and service worker are separate execution contexts. They share *nothing* in memory. Use `chrome.runtime.sendMessage()` and `chrome.runtime.onMessage` with typed message objects.
- **`chrome.storage` is the single source of truth.** Never cache state in a variable in the background service worker. it gets wiped when the worker terminates. Read from and write to `chrome.storage.local` (or `.sync` for cross-device).
- **Content scripts run in the webpage's DOM.** They can read and modify the page but cannot access extension APIs directly (except messaging and storage). Keep content scripts as small as possible. inject only what's needed.
- **Popup is ephemeral.** It mounts when clicked, unmounts when closed. Never put long-running logic in the popup. delegate to the service worker via messages.

## Coding Conventions

- **All messages are typed.** Define a union type in `shared/messages.ts`: `type Message = { type: 'GET_DATA'; payload: string } | { type: 'SET_CONFIG'; payload: Config }`. Never use untyped `sendMessage({ action: 'doThing' })`.
- **Storage access is wrapped.** Use a typed wrapper in `shared/storage.ts` that handles serialization and provides get/set per key. Never call `chrome.storage.local.get()` with raw string keys. the wrapper enforces types.
- **Content script CSS is scoped.** Use unique class prefixes (`ext-myext-`) or Shadow DOM to prevent style collisions with the host page. Never inject global styles that could break the page.
- **Service worker is event-only.** Register all listeners at the top level (not inside async callbacks). Chrome requires listeners to be registered synchronously on worker startup.
- **Permissions are minimal.** Request only the permissions you need in `manifest.json`. Use `optional_permissions` for features the user might not use. Chrome Web Store reviewers reject extensions with over-broad permissions.

## Library Preferences

- **Build tool:** Vite + CRXJS (`@crxjs/vite-plugin`). handles HMR, manifest generation, and content script bundling. Not webpack (CRXJS is purpose-built for extensions).
- **UI framework:** React for complex popup/options UIs. Vanilla TypeScript for simple popups. Not Svelte or Vue. React has the best CRXJS support.
- **Storage:** `chrome.storage.local` with a typed wrapper. Not localStorage. it's not accessible from service workers, and `chrome.storage` syncs across contexts.
- **HTTP requests:** `fetch` from the service worker (it has full network access). Content scripts should delegate network requests to the service worker. they inherit the page's CORS restrictions.

## NEVER DO THIS

1. **Never rely on persistent state in the service worker.** It can be terminated and restarted at any time. All state must be read from `chrome.storage` on every event. Module-level `let currentUser = null` will be `null` after worker restart.
2. **Never register event listeners inside async callbacks.** `chrome.runtime.onMessage.addListener()` must be called synchronously at the top level of the service worker. Registering inside `setTimeout` or `await` means the listener may not exist when the worker starts.
3. **Never inject global CSS from content scripts without scoping.** Your `.container { display: flex }` will break the host page. Prefix all classes (`ext-myext-container`) or use Shadow DOM.
4. **Never request `<all_urls>` permission unless absolutely necessary.** Chrome Web Store will flag it. Use specific host patterns: `"https://*.example.com/*"`. Add `optional_permissions` for user-activated expansion.
5. **Never use `eval()` or inline scripts.** MV3's Content Security Policy blocks both. All JavaScript must be in separate files. No `onclick="handler()"` in HTML. use `addEventListener`.
6. **Never assume the popup stays open.** Users click away constantly. If a process takes more than a second, send it to the service worker and display results when the popup reopens by reading from storage.
7. **Never send sensitive data through content script messaging without validation.** Content scripts run in the page context. a malicious page can spoof messages. Always validate `sender.id` in the service worker's message handler.

## Testing

- Use Vitest for unit tests on shared modules (message handlers, storage wrappers, utilities).
- Use Playwright with the Chrome Extension testing fixture for E2E tests. load the unpacked extension in a test browser.
- Test content scripts by mounting them in a JSDOM environment with a mock DOM.
- Test the service worker by mocking `chrome.*` APIs with `jest-chrome` or a custom mock.
