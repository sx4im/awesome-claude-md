## Architecture Rules

- Use `createClient()` from `@libsql/client` with `url` and `authToken` for remote Turso connections
- For edge deployments, use embedded replicas: `createClient({ url: "file:local.db", syncUrl: tursoUrl, authToken })` with periodic `db.sync()`
- All database access goes through Drizzle ORM query builder; never write raw SQL strings except in migrations
- Batch operations use `client.batch()` for atomic multi-statement transactions (Turso-specific API)
- Drizzle schema uses SQLite types: `text()`, `integer()`, `real()`, `blob()`; no `varchar` or `boolean` (use integer 0/1)
- Foreign keys require `PRAGMA foreign_keys = ON` explicitly; libSQL does not enable it by default
- Connection pooling is handled by Turso's HTTP protocol; do not implement client-side pooling

## Coding Conventions

- Define all tables in `schema.ts` using `sqliteTable()` from drizzle-orm/sqlite-core
- Export inferred types: `type User = typeof users.$inferSelect` and `type NewUser = typeof users.$inferInsert`
- Query functions are async, accept Drizzle db instance as first parameter for testability
- Use `db.select().from(users).where(eq(users.id, id))` pattern; avoid raw SQL in application code
- Timestamps stored as ISO 8601 text strings (SQLite has no native datetime); use `text("created_at").default(sql`(datetime('now'))`)
- Environment variables: `TURSO_DATABASE_URL`, `TURSO_AUTH_TOKEN`, `TURSO_SYNC_INTERVAL_MS`
- JSON data stored as text columns with `JSON.parse/stringify` in application code; use Drizzle's `.$type<MyType>()` for type narrowing

## Library Preferences

- @libsql/client over better-sqlite3 for Turso compatibility and embedded replica support
- Drizzle ORM over Prisma for SQLite edge compatibility and lighter bundle size
- drizzle-kit over manual migration files for schema diffing and migration generation
- Hono over Express for edge-deployed APIs (Cloudflare Workers, Vercel Edge)
- drizzle-zod for generating Zod validation schemas from Drizzle table definitions
- Turso CLI (`turso`) for database management, not the web dashboard for scripted operations

## File Naming

- Schema files: camelCase `.ts` in `db/` directory
- Query modules: camelCase `.ts` matching the domain entity, e.g., `users.ts`, `posts.ts`
- Migration files: auto-generated by drizzle-kit with timestamp prefix
- Config files: camelCase with `.config.ts` suffix
- Scripts: kebab-case `.ts` or `.sh` in `scripts/`

## NEVER DO THIS

1. Never hardcode Turso database URLs or auth tokens; always use environment variables loaded at startup
2. Never use `client.execute()` for multiple related writes; use `client.batch()` for atomic transactions
3. Never rely on SQLite autoincrement for distributed ID generation; use ULIDs or UUIDs as text primary keys
4. Never forget to call `db.sync()` on embedded replicas; stale reads will occur without periodic sync
5. Never use `PRAGMA journal_mode` or `PRAGMA wal` commands; Turso manages journaling at the server level
6. Never assume Turso supports all SQLite extensions; check compatibility for FTS5, JSON1, and others

## Testing

- Unit test query functions by creating an in-memory libSQL client: `createClient({ url: ":memory:" })`
- Run Drizzle migrations against the in-memory database before each test suite
- Integration tests use a dedicated Turso test database created per CI run via `turso db create test-<sha>`
- Clean up test databases in CI teardown: `turso db destroy test-<sha>`
- Test embedded replica sync by writing to remote and asserting data appears locally after `sync()`
- Validate schema migrations are reversible; drizzle-kit generates `up` SQL, write `down` SQL manually
- Test batch transaction atomicity: verify all statements roll back if any single statement fails
- Mock `@libsql/client` in unit tests using vitest's `vi.mock()` for isolated query function testing
