## Architecture Rules

- **HTMX controls the front-end.** Avoid writing custom JavaScript. Instead, use HTMX attributes (`hx-get`, `hx-post`, `hx-target`, `hx-swap`) on HTML elements. The server responds with HTML fragments, not JSON.
- **Templ for type-safe components.** We use Templ (`.templ` files) to generate Go code from HTML. Never use strings or standard `html/template` package to build HTML responses. Templ guarantees compile-time safety and integrates directly with Go handlers.
- **The "Boring Architecture."** Your Go backend generates HTML. No React, no Vite, no SPAs, no heavy JSON APIs. The client executes zero complex JS. State lives in the database.
- **Standard Layouts.** Define a base HTML layout containing the `<head>`, Tailwind classes, and htmx script tags. Handlers render partials or wrap partials in the base layout depending on whether the request is an HTMX request or a full-page load.
- **Locality of Behavior (LoB).** Keep behavior close to elements. If a button clicks and replaces a list, the button defines `hx-get="/list"` and `hx-target="#list-container"`. Do not create abstract JS files separating behavior from structure.

## Coding Conventions

- **Makefiles run the stack.** Use Makefiles to tie together `templ generate`, `npx tailwindcss`, and `go build`. Never build manually.
- **HTTP Handlers return Components.** A standard handler fetches data from `internal/database/db.go`, constructs a `templ.Component`, and calls `component.Render(ctx, w)`. 
- **Context passing.** Pass the `context.Context` everywhere (down to templ components, database calls). It tracks request cancellation.
- **`sqlc` for Database.** Write explicit `query.sql` files. Run `sqlc generate`. Never use heavy ORMs like GORM. Use compile-time evaluated structs generated purely from raw PostgreSQL / SQLite queries.
- **Detect HTMX requests.** Use middlewear or inspect `r.Header.Get("HX-Request") == "true"`. If standard request, render the layout + content. If HX request, render only the specific content fragment to swap.

## Library Preferences

- **Router:** Chi (stdlib native) or Echo (slightly higher level). Avoid heavy frameworks like Gin which don't map naturally to returning standard `http.HandlerFunc` components.
- **Template Engine:** Templ (`a-h/templ`). Not standard `html/template`.
- **Database:** `database/sql` + `sqlc`. Not GORM, not Ent (unless building massive systems).
- **CSS:** Tailwind CSS. Use `tailwind CLI` in the Makefile to watch `.templ` files and output to `static/css/styles.css`.
- **Hot Reloading:** `Air` (Live-reloading tool for Go apps). Your air config should trigger both Tailwind and Templ generation before compiling Go.

## NEVER DO THIS

1. **Never return JSON for an endpoint if HTMX is consuming it.** HTMX expects HTML. The entire paradigm is Hypermedia As The Engine of Application State. Handlers return `.templ` fragments.
2. **Never build a full SPA-like router in JavaScript.** HTMX handles pushing URL history via `hx-push-url="true"`. Let the backend handle routing using standard Go HTTP multiplexers.
3. **Never write raw strings for HTML.** Writing `w.Write([]byte("<div>hello</div>"))` defeats type safety and escaping rules. Construct a Templ component explicitly.
4. **Never forget CSRF protection.** Forms submitted by HTMX still need CSRF tokens. Usually added via HTMX request headers config or a meta tag standard.
5. **Never write logic inside `.templ` files.** Templ looks like React/JSX, but it compiles to Go strings. Use standard `if / else` and `for` loops inside the templ file, but never write database access or complex data transformations in the template. Pass pre-computed data structures to the component.
6. **Never use an ORM when sqlc applies.** Writing raw SQL queries is safer because `sqlc` generates typed Go boilerplate. The query is validated against your SQL schema file instantly.
7. **Never ignore HTTP cancellation requests.** If the user clicks away or HTMX aborts a request, the `r.Context()` drops. Pass it directly to database calls so expensive queries cancel immediately.

## Testing

- Create an integration test suite targeting HTTP handlers (`httptest.NewRecorder()`). Verify the returned HTML contains specific substrings or attributes.
- Because the backend *is* the frontend, your standard HTTP integration tests basically amount to testing the entire UI rendering flow concurrently.
- No need for Jest/Playwright unless validating highly complex DOM drag-and-drop or Alpine.js edgecases.
