# [PROJECT NAME] - [ONE LINE DESCRIPTION]

## Tech Stack

- TypeScript 5.4+ with strict mode
- Neon serverless PostgreSQL as the primary database
- @neondatabase/serverless driver for HTTP and WebSocket connections
- Drizzle ORM for type-safe schema definitions and queries
- drizzle-kit for migration management
- Node.js 20 LTS or edge runtimes (Vercel Edge, Cloudflare Workers)
- Neon CLI for branch management and database operations

## Architecture Rules

- Use `neon()` from `@neondatabase/serverless` for HTTP-based queries in serverless functions (one-shot, no persistent connection)
- Use `Pool` from `@neondatabase/serverless` with WebSocket for long-lived connections in Node.js servers
- Pass the Neon client to Drizzle: `drizzle(neon(connectionString))` for HTTP or `drizzle(pool)` for WebSocket mode
- Every PR creates a Neon branch via `neon branches create --project-id <id> --parent main`; preview deployments use branch connection string
- Main branch is the production database; feature branches are instant copy-on-write forks for dev/preview
- Use Drizzle migrations for schema changes; apply to the Neon main branch, branches inherit schema from parent
- Connection strings follow format: `postgresql://user:pass@ep-xxx.region.neon.tech/dbname?sslmode=require`

## Coding Conventions

- Define tables using `pgTable()` from drizzle-orm/pg-core with PostgreSQL-native types
- Use `uuid("id").defaultRandom().primaryKey()` for primary keys, not serial integers
- Timestamps use `timestamp("created_at", { withTimezone: true }).defaultNow()`; always store with timezone
- Export inferred types: `type User = typeof users.$inferSelect` and `type InsertUser = typeof users.$inferInsert`
- Query functions accept a `db` parameter typed as `NeonHttpDatabase` for testability and connection flexibility
- Use `db.transaction(async (tx) => { ... })` for multi-statement atomic operations
- Connection string from `DATABASE_URL` env var; branch URLs override via `NEON_BRANCH_URL` in preview deployments
- Use `drizzle-zod` to generate Zod schemas from table definitions for API input validation

## Library Preferences

- @neondatabase/serverless over pg or postgres.js for Neon-optimized connections (HTTP fetch driver)
- Drizzle ORM over Prisma for edge runtime compatibility and SQL-like query syntax
- drizzle-kit over raw SQL migration files for type-safe schema diffing
- Neon CLI over web console for scripted branch management in CI/CD
- drizzle-zod for request validation schemas derived from database tables
- kysely as alternative query builder when raw SQL control is needed alongside Drizzle

## File Naming

- Schema files: `schema.ts` for all tables, split into `schema/users.ts` etc. if exceeding 200 lines
- Query modules: camelCase `.ts` per domain entity in `queries/`
- Migration files: auto-generated by drizzle-kit with sequential numbering
- Config: `drizzle.config.ts` at project root
- Branch management scripts: kebab-case `.sh` in `scripts/`

## NEVER DO THIS

1. Never use persistent connection pools in serverless functions; use `neon()` HTTP driver which uses fetch under the hood
2. Never connect without `sslmode=require` in the connection string; Neon requires TLS for all connections
3. Never run migrations directly against production main branch without testing on a Neon branch first
4. Never leave Neon branches running after PR merge; they consume compute resources even when idle
5. Never use `serial` primary keys in distributed/branched environments; use `uuid` to avoid ID conflicts across branches
6. Never skip the `{ withTimezone: true }` option on timestamp columns; timezone-naive timestamps cause production bugs
7. Never hardcode the Neon project ID or branch ID; use environment variables and Neon CLI for discovery

## Testing

- Unit tests use a dedicated Neon branch created in CI setup: `neon branches create --name test-<sha>`
- Run migrations against the test branch before test execution
- Use `db.transaction()` with rollback for test isolation: wrap each test in a transaction that never commits
- Test query functions by injecting the Neon database instance; mock with `vi.fn()` for pure unit tests
- Integration tests verify Drizzle query results against known seed data loaded via `seed.ts`
- Test branching workflow: create branch, apply migration, verify schema, delete branch
- CI pipeline deletes test branches in the teardown step regardless of test outcome
- Validate connection handling: test that HTTP driver correctly handles cold starts and connection reuse
- Use Neon's SQL Editor in the dashboard for ad-hoc query debugging during development
