## Architecture Rules

- **Use Component-like PHP structure.** The `functions.php` file should strictly be an index that `require_once`'s isolated files inside the `inc/` directory. Do not write 5,000 lines of functional logic in a single file.
- **`theme.json` for Global Styles.** Rely on WordPress's `theme.json` to define typography, color palettes, and layout spacing instead of hardcoding them in CSS. This explicitly wires the theme up to the Gutenberg Site Editor securely.
- **Enqueue correctly.** All CSS and JavaScript must be loaded via `wp_enqueue_style` and `wp_enqueue_script` inside an action hooked to `wp_enqueue_scripts`. Never print `<link>` or `<script>` tags directly in the `<head>` of `header.php`.
- **Vanilla JavaScript.** Modern WordPress does not require jQuery. Drop the dependency, write native ES6+, and compile it down using `@wordpress/scripts`.
- **Sanitization & Escaping (SECURE EVERYTHING).** Late escaping is mandatory. Whenever printing data to the browser, it must be passed through `esc_html()`, `esc_attr()`, or `esc_url()`. When saving data to the database, use `sanitize_text_field()` or equivalent mapping.

## Coding Conventions

- **Prefix everything.** If your theme is named "Neon", prefix all custom functions, constants, and global variables with `neon_`. `neon_setup_theme()`, `neon_enqueue_scripts()`. This prevents fatal collisions with plugins.
- **Action & Filter hooks are the API.** To modify WordPress behavior, you `add_action` or `add_filter`. Do not edit core files. Do not hack standard behaviors directly.
- **Use Template Parts.** `get_template_part('parts/content', 'article')`. DRY up your templates. Don't copy/paste the loop structure across `archive.php`, `index.php`, and `category.php`.
- **WP_Query meticulously.** Always call `wp_reset_postdata()` immediately after completing a custom `new WP_Query()` loop. Failing to do this breaks the global `$post` object for the rest of the page.
- **Block-based mindset.** For client editability, build custom blocks (using ACF Blocks or native `@wordpress/blocks`). Do not rely on massive, fragile Meta Box (post meta) arrays to build complex page layouts anymore.

## Library Preferences

- **Build Tool:** `@wordpress/scripts` (WebPack wrapper tuned specifically for WP and Gutenberg).
- **Custom Data:** Advanced Custom Fields (ACF) Pro. Essential for rapid block creation and option pages. Native Gutenberg is fine if fully committed to React editing.
- **Dev Environment:** Local (by Flywheel) or `@wordpress/env` (Docker).
- **CSS:** Tailwind CSS. It integrates fine within WordPress development, outputting to `build/style.css` which is then enqueued.

## NEVER DO THIS

1. **Never use `extract()`.** `extract()` creates variables natively in the current scope from an array. It makes it impossible to trace variable origins. Explicitly declare variables cleanly.
2. **Never query the database natively without `$wpdb->prepare()`.** If you must write a raw SQL query (which is rare, use `WP_Query` instead), it must be parameterized via `$wpdb->prepare()` to prevent SQL injection.
3. **Never `echo` directly from the database.** Always `echo esc_html($value)`. If it's a known URL, `echo esc_url()`. If it has safe HTML elements (like `<strong>` or `<a>`), use `wp_kses_post()`. 
4. **Never deregister Core WordPress jQuery unless you know exactly what you are doing.** Plugins rely on it. If you dequeue it globally, a form plugin or a slider plugin will silently break.
5. **Never put Custom Post Types or Taxonomies in the theme if they contain critical business logic.** If the client switches themes, they lose their data. `inc/post-types.php` is fine for display-centric CPTs (like "Sliders"), but core CPTs ("Real Estate Listings") belong in a custom site plugin.
6. **Never leave debug mode on in production.** `WP_DEBUG`, `WP_DEBUG_LOG`, and `WP_DEBUG_DISPLAY` ensure you see errors in development, but must correctly be mapped in `wp-config.php` for production deployments.
7. **Never write static URLs.** `href="/wp-content/themes/neon/assets/img.png"` will break when migrating from local to staging to production. Use `echo get_template_directory_uri() . '/assets/img.png'`.

## Testing

- **WP-CLI.** Use it locally. `wp theme check` and `wp core update`. 
- **Theme Check Plugin.** Install this on a dev instance. It statically analyzes your code against WordPress standards for missing `wp_footer()` calls, hardcoded URLs, and missing hooks.
- **PHPCS.** Use `phpcs` with the `WordPress-Coding-Standards` (WPCS) ruleset. It catches missing sanitization statically before you push.
