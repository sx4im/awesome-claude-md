## Architecture Rules

- **Convex functions are the backend.** All server logic lives in the `convex/` directory as queries, mutations, or actions. Never create Next.js API routes for data operations.
- **Queries are reactive and read-only.** `query` functions automatically re-run when underlying data changes. They MUST be deterministic and side-effect free. Never write data, call external APIs, or use `Date.now()` inside a query.
- **Mutations are transactional writes.** `mutation` functions read and write data atomically. They MUST be deterministic. Never call external APIs or use non-deterministic functions inside mutations.
- **Actions are for side effects.** `action` functions can call external APIs, use randomness, and perform I/O. They CANNOT directly read or write the database. To write data from an action, call a mutation via `ctx.runMutation`.
- **Schema defines all tables upfront.** Every table and its fields must be declared in `convex/schema.ts`. Convex enforces the schema at runtime. Never write documents with fields not in the schema.

## Coding Conventions

- **Use `v` validators everywhere.** Define argument validators and schema fields with `v.string()`, `v.number()`, `v.id("tableName")`, etc. Never use `any` or skip argument validation.
- **Reference documents by `Id<"tableName">`.** Convex uses typed document IDs. Never store IDs as plain strings. Use `v.id("users")` in validators and `Id<"users">` in TypeScript types.
- **Client data fetching uses hooks.** Use `useQuery(api.feature.functionName, args)` in components. It returns `undefined` while loading, then the value. Never use `useEffect` + manual fetching.
- **Paginated queries use `.paginate(opts)`.** Return `{ page, isDone, continueCursor }` from queries. On the client, use `usePaginatedQuery` with `{ numItems }`. Never use skip/offset pagination.
- **File naming in convex/.** One file per feature domain (`convex/messages.ts`, `convex/users.ts`). Each file exports multiple query/mutation/action functions. Do not create one file per function.

## Library Preferences

- **Auth:** Convex Auth or [Clerk] integration. Use the official Convex adapter; never roll custom session management.
- **File Storage:** Convex built-in file storage (`ctx.storage`). Never use S3 or external storage for user uploads.
- **Search:** Convex built-in full-text search indexes. Define in schema with `searchIndex`. Never pull in Algolia for simple text search.
- **Scheduling:** Convex `crons.ts` and `ctx.scheduler.runAfter()`. Never use external cron services for Convex-triggered jobs.
- **Styling:** Tailwind CSS. Component library: [shadcn/ui].

## NEVER DO THIS

1. **Never edit files in `convex/_generated/`.** These are auto-generated by `npx convex dev`. Manual edits are overwritten immediately and will cause type errors.
2. **Never call external APIs from queries or mutations.** Only `action` functions can perform I/O. Calling `fetch` in a query or mutation will throw a runtime error.
3. **Never use `Date.now()` or `Math.random()` in queries or mutations.** These are non-deterministic. Use `ctx.now()` for timestamps in mutations. For randomness, generate it in an action and pass it to a mutation.
4. **Never store Convex document IDs as strings.** Always use `v.id("tableName")` in schemas and `Id<"tableName">` in types. String comparison on IDs will produce incorrect results.
5. **Never use Next.js `fetch` with `revalidate` for Convex data.** Convex queries are inherently reactive. Using ISR/SSG caching on top defeats real-time updates and causes stale data.
6. **Never create indexes after the fact without migration.** Define indexes in `schema.ts` before querying by those fields. Queries without indexes scan the full table and will be slow or error at scale.
7. **Never return entire tables from queries.** Always filter and paginate. Convex charges for bandwidth, and large unfiltered reads will hit function time limits.

## Testing

- **Backend function tests:** Use Convex's testing utilities. Create a `ConvexTestingHelper` that runs functions against an in-memory backend. Test queries and mutations in isolation.
- **Component tests:** Mock Convex hooks with `vi.mock("convex/react")`. Provide deterministic return values for `useQuery` and `useMutation`. Test loading and error states by returning `undefined` and throwing.
- **Integration tests:** Use `npx convex run` to invoke functions against the dev deployment. Verify data integrity across mutation + query sequences.
- **E2E tests:** Use [Playwright/Cypress] against a dedicated Convex dev deployment. Never run E2E against production.
- Run tests: `npx vitest` for unit/component, `npx convex run` for backend smoke tests.
