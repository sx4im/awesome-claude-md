## Architecture Rules

- **`_helpers.tpl` is the template library.** All reusable named templates (`define`) live here: labels, selectors, names, image references. Templates in resource files call `include` to get computed values. Never duplicate label logic across files.
- **Values are the only interface.** Users configure the chart exclusively through `values.yaml` overrides. Every configurable aspect (replicas, resources, image, ports, feature toggles) is a value. Never require users to edit templates. If something needs customization and is not in values, the chart is incomplete.
- **Sensible defaults, zero required values.** `helm install myapp ./chart` must produce a working deployment with zero value overrides. Required values (like image repository) have defaults that point to a functional image. Document which values should be overridden for production.
- **Feature flags for optional resources.** Ingress, HPA, PDB, ServiceMonitor, and NetworkPolicy are enabled/disabled via boolean values: `ingress.enabled: false`. The template wraps the entire file in `{{- if .Values.ingress.enabled }}`. Never force users to deploy resources they don't need.
- **Dependency charts via Chart.yaml.** Third-party dependencies (Redis, PostgreSQL, RabbitMQ) are declared in `Chart.yaml` under `dependencies:` with `condition:` flags. Never embed another chart's templates directly. Run `helm dependency update` to pull them into `charts/`.

## Coding Conventions

- **Named templates for labels and selectors.** Define `chart.labels`, `chart.selectorLabels`, and `chart.fullname` in `_helpers.tpl`. Every resource uses `{{- include "chart.labels" . | nindent 4 }}` for metadata labels. Selector labels are a subset of full labels — selectors are immutable, so never include chart version in selector labels.
- **Quote strings, don't quote integers and booleans.** `port: {{ .Values.service.port }}` (no quotes for int). `name: {{ .Values.name | quote }}` (quote strings). Kubernetes rejects a string where an integer is expected. `"8080"` is not the same as `8080` in a container port spec.
- **Use `nindent` not `indent`.** `{{- include "chart.labels" . | nindent 4 }}` trims the preceding whitespace and indents correctly. `indent` without the leading dash produces blank lines and broken YAML. This is the most common Helm template rendering bug.
- **Default values with `default`.** `{{ .Values.nodeSelector | default dict | toYaml | nindent 8 }}` handles both "key exists but is empty" and "key is missing." Never assume a nested value path exists without defaulting.
- **Image reference as a helper.** Define `{{- define "chart.image" -}}{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}{{- end }}`. Centralizes image logic including registry prefix and digest support.

## Library Preferences

- **Template functions:** Sprig library (bundled with Helm). Use `default`, `required`, `toYaml`, `nindent`, `trimSuffix`, `b64enc`. Not custom Go template functions.
- **Schema validation:** `values.schema.json` using JSON Schema draft-07. Validates values before rendering templates. Catches typos and type errors at `helm install` time.
- **Documentation:** helm-docs to auto-generate README from values.yaml comments. Not manually written docs that drift from actual values.
- **Testing:** helm-unittest for template assertion tests. ct (chart-testing) for lint and install tests in CI.
- **Linting:** `helm lint` and `ct lint` in CI. Catches template syntax errors, missing required values, and chart metadata issues.

## File Naming

- Templates: `resource-type.yaml` matching the Kubernetes resource kind (e.g., `deployment.yaml`, `service.yaml`)
- Helpers: `_helpers.tpl` (underscore prefix means Helm does not render it as a resource)
- Test values: `ci/descriptive-name-values.yaml`
- Chart tests: `templates/tests/test-description.yaml`
- One resource per file. Never put Deployment and Service in the same template file.

## NEVER DO THIS

1. **Never use `helm install` without `--namespace`.** Omitting the namespace deploys to `default`. Production workloads in `default` namespace break RBAC boundaries, make cleanup dangerous, and conflict with other installs. Always specify and create the namespace: `--namespace myapp --create-namespace`.
2. **Never use `lookup` in templates for required resources.** `lookup` queries the live cluster during `helm template` but returns empty when rendering offline (CI, diff, dry-run). Templates that depend on `lookup` break in any non-connected context. Use values or Chart dependencies instead.
3. **Never hardcode the release name in templates.** Use `{{ .Release.Name }}` and `{{ include "chart.fullname" . }}`. Hardcoded names prevent installing multiple releases of the same chart and cause resource naming collisions.
4. **Never use `toYaml` without `nindent`.** `{{ toYaml .Values.resources }}` dumps YAML at column 0 regardless of its position in the template. The result is invalid YAML. Always pipe through `nindent`: `{{ toYaml .Values.resources | nindent 12 }}`.
5. **Never put logic in NOTES.txt that differs from templates.** If NOTES.txt constructs a URL using different logic than `ingress.yaml`, users get wrong instructions. NOTES.txt should use the same `_helpers.tpl` templates as the resource files.
6. **Never use `required` without a clear error message.** `{{ required "" .Values.image.repository }}` produces a cryptic error. Use `{{ required "image.repository is required" .Values.image.repository }}` so users know exactly what to set.
7. **Never bump `version` without bumping `appVersion` when the application changes.** `version` is the chart version (packaging changes). `appVersion` is the application version (what is deployed). Deploying app v2.0 via chart v1.0.0 with `appVersion: 1.0` misleads users about what is running. Keep them synchronized.

## Testing

- Run `helm lint .` and `helm template . --debug` on every PR. Lint catches schema violations and deprecated API versions. Template renders the full output without a cluster, exposing rendering errors.
- Use helm-unittest to assert specific resource properties: verify Deployment has correct image, Service has correct port, Ingress has correct host. Tests live in `tests/` and run in CI without a cluster.
- Use ct (chart-testing) `ct lint-and-install` with a kind cluster in CI. This performs a real `helm install`, waits for pods to be ready, runs `helm test`, then cleans up. Catches runtime issues that template rendering misses.
- Maintain multiple values files in `ci/` that exercise different feature combinations: default values, all features enabled, minimal values, production-like values. Each is a separate lint and install test case.
- Test chart upgrades: install the previous chart version, then upgrade to the new version. Helm upgrade failures (immutable field changes, missing migration hooks) are the most common production issues and are only caught by testing the upgrade path.
